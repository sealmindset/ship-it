# DevOps Onboarding Guide

How to configure `/ship-it` for your organization. You do this once, then every app built by `/make-it` (or any developer) can deploy through the same pipeline.

## What You're Setting Up

`/ship-it` uses a file called `.ship-it.yml` in each repo. It has three sections:

| Section | Who fills it | What it controls |
|---------|-------------|-----------------|
| `app` | Auto-generated by /make-it (or the developer) | What's being deployed -- name, stack, services |
| `infra` | **You (DevOps)** | Where to deploy -- cloud account, networking, clusters, DNS |
| `deployment` | **You (DevOps)** | How the pipeline behaves -- environments, reviewers, strategy |

The `app` section is already filled when you get the PR. Your job is the `infra` and `deployment` sections.

## Two Approaches

### Option A: Org-Wide Template (Recommended)

Create a single `.ship-it.yml` template with your org's infrastructure values pre-filled. Every new app gets a copy with the `app` section swapped out.

Best for: Teams where all apps deploy to the same cluster, VPC, and DNS zone.

### Option B: Per-App Configuration

Fill in `.ship-it.yml` individually for each app. The developer (or /make-it) creates the PR, and you add the `infra` section before the first deployment.

Best for: Teams where apps have different infrastructure requirements.

## Setup Steps

### 1. Choose your cloud provider

Copy the matching template into your org's config repo or directly into app repos:

- **AWS**: `templates/ship-it-aws.yml`
- **Azure**: `templates/ship-it-azure.yml`

### 2. Fill in the infrastructure section

Each field has a comment explaining what it is and where to find the value. Here's what you need:

#### AWS

| Field | Where to find it | Example |
|-------|-----------------|---------|
| `account_id` | AWS Console > top-right > Account ID | `123456789012` |
| `region` | Your primary region | `us-east-1` |
| `ecr_registry` | `{account_id}.dkr.ecr.{region}.amazonaws.com` | `123456789012.dkr.ecr.us-east-1.amazonaws.com` |
| `vpc_id` | VPC Console > Your VPC | `vpc-0abc123def456` |
| `private_subnets` | VPC Console > Subnets (filter: private) | `["subnet-abc123", "subnet-def456"]` |
| `public_subnets` | VPC Console > Subnets (filter: public) | `["subnet-ghi789", "subnet-jkl012"]` |
| `ecs.cluster_name` | ECS Console > Clusters (or create one) | `apps-cluster` |
| `ecs.execution_role_arn` | IAM Console > Roles > ecsTaskExecutionRole | `arn:aws:iam::123456789012:role/ecsTaskExecutionRole` |
| `ecs.task_role_arn` | IAM Console > Roles > ecsTaskRole | `arn:aws:iam::123456789012:role/ecsTaskRole` |
| `alb.certificate_arn` | ACM Console > Certificates | `arn:aws:acm:us-east-1:123456789012:certificate/abc-123` |
| `dns.hosted_zone_id` | Route53 Console > Hosted Zones | `Z1234567890ABC` |
| `dns.domain` | Your apps domain | `apps.example.com` |

#### Azure

| Field | Where to find it | Example |
|-------|-----------------|---------|
| `subscription_id` | Azure Portal > Subscriptions | `12345678-abcd-efgh-ijkl-123456789012` |
| `resource_group_prefix` | Your naming convention | `rg` (creates `rg-{slug}-dev`, `rg-{slug}-prod`) |
| `acr_name` | Azure Portal > Container Registries | `myorgacr` |
| `acr_login_server` | ACR overview page | `myorgacr.azurecr.io` |
| `aks_cluster` | Azure Portal > Kubernetes services | `apps-aks-cluster` |
| `aks_resource_group` | Resource group containing AKS | `rg-shared-aks` |
| `vnet_name` | Azure Portal > Virtual networks | `apps-vnet` |
| `vnet_resource_group` | Resource group containing VNet | `rg-shared-networking` |
| `dns_zone` | Azure Portal > DNS zones | `apps.example.com` |
| `dns_resource_group` | Resource group containing DNS zone | `rg-shared-dns` |
| `key_vault_prefix` | Your naming convention | `kv` (creates `kv-{slug}-dev`) |

### 3. Configure the deployment section

```yaml
deployment:
  # GitHub environment names -- must match what's in repo Settings > Environments
  environments:
    dev: dev
    staging: staging              # Remove if you don't use staging
    production: production

  # Who reviews production PRs (GitHub usernames)
  reviewers:
    - your-devops-lead
    - your-team-lead

  # Go-live checklist -- customize for your org
  prerequisites:
    - "Does this app need users to log in? (SSO / App registration)"
    - "Does this need a secure web address? (SSL certificate)"
    - "Who should have access in production? (Permissions / RBAC)"
    - "Does this need a URL people can visit? (DNS setup)"
    - "Does this need to talk to internal systems? (Network / firewall)"

  # If your org has a shared reusable workflow, reference it here
  # reusable_workflow: "my-org/shared-workflows/.github/workflows/ship-it-pipeline.yml@main"

  strategy: rolling               # rolling, blue-green, canary
  rollback: true
```

### 4. Set up GitHub environments

In each repo (or as an org default):

1. Go to **Settings > Environments**
2. Create `dev` environment (no protection rules needed)
3. Create `production` environment with:
   - **Required reviewers**: Add the same people from `deployment.reviewers`
   - **Deployment branches**: Limit to `main` only

### 5. Set up GitHub secrets

Each environment needs these secrets:

#### AWS
| Secret | Environment | Value |
|--------|-------------|-------|
| `AWS_ACCESS_KEY_ID` | dev, production | IAM user or OIDC role credentials |
| `AWS_SECRET_ACCESS_KEY` | dev, production | IAM user or OIDC role credentials |
| `AWS_REGION` | dev, production | e.g., `us-east-1` |

Or better -- use [OIDC federation](https://docs.github.com/en/actions/deployment/security-hardening-your-deployments/configuring-openid-connect-in-amazon-web-services) to avoid long-lived credentials.

#### Azure
| Secret | Environment | Value |
|--------|-------------|-------|
| `AZURE_CLIENT_ID` | dev, production | Service principal app ID |
| `AZURE_TENANT_ID` | dev, production | Azure AD tenant ID |
| `AZURE_SUBSCRIPTION_ID` | dev, production | Subscription ID |

Use [OIDC federation](https://docs.github.com/en/actions/deployment/security-hardening-your-deployments/configuring-openid-connect-in-microsoft-azure) with federated credentials.

### 6. (Optional) Shared reusable workflow

If you want all apps to use the same pipeline, create a reusable workflow in a shared repo:

```
my-org/shared-workflows/.github/workflows/ship-it-pipeline.yml
```

Then set `deployment.reusable_workflow` in `.ship-it.yml`. Each app gets a thin caller workflow that references your shared pipeline. This gives you centralized control over the deployment process.

## How It All Connects

```
Developer runs /ship-it
       |
       v
Reads .ship-it.yml + app-context.json
       |
       v
Merges: .ship-it.yml > app-context > auto-detect > defaults
       |
       v
Generates workflow based on infra section:
  - infra populated  -> real deploy steps (Docker build, ECR push, ECS deploy)
  - infra empty      -> placeholder steps ("pending DevOps configuration")
  - reusable_workflow -> thin caller referencing your shared pipeline
       |
       v
Creates PR with:
  - App details from app-context.json
  - Infrastructure status (configured or pending)
  - Smart prerequisites checklist
  - Intent label (experiment/shareable/prod-ready)
       |
       v
Pipeline runs on PR merge:
  - experiment  -> CI only, no deploy
  - shareable   -> deploy to dev
  - prod-ready  -> deploy to dev, then prod (after approval)
```

## Validating Your Setup

After filling in `.ship-it.yml`, test with a sample app:

```bash
# Create a test repo with a simple app
mkdir test-app && cd test-app
git init && echo '{"name":"test"}' > package.json
echo 'console.log("hello")' > index.js

# Copy your .ship-it.yml into the repo
cp /path/to/your/ship-it.yml .ship-it.yml

# Run /ship-it
claude
/ship-it
```

Verify:
- The PR body shows "DevOps infrastructure configured" (not "pending")
- The generated workflow has real deploy steps (not placeholder echo commands)
- The correct reviewers are assigned
- Labels are applied

## Updating Infrastructure

When infrastructure changes (new cluster, new region, new DNS zone):

1. Update `.ship-it.yml` in each affected repo
2. Or update your org-wide template and propagate to repos
3. Run `/ship-it` again on any open PRs -- it regenerates the workflow

`/ship-it` never overwrites existing workflow files. If you need to regenerate, delete `.github/workflows/ship-it.yml` first.

## Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| PR says "pending DevOps configuration" | `infra` section is empty | Fill in the `infra` section of `.ship-it.yml` |
| Workflow has placeholder deploy steps | Same as above | Same as above |
| Deployment fails with auth error | GitHub environment secrets not set | Add cloud credentials to the environment in repo Settings |
| Reviewer not assigned | Username not in `deployment.reviewers` | Add their GitHub username to the list |
| Wrong environment name | Mismatch between `.ship-it.yml` and repo Settings | Ensure `deployment.environments` values match GitHub environment names exactly |
